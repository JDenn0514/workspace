## Process Record

### Proposal

**From `spec.md` — algorithm design:**

`pvm()` implements the Percentage Valuation Method in three conceptual layers:

1. **Above-replacement contribution.** For each rostered player and scored category, compute `contrib = PS[i,c] - RS[c]` (higher-is-better) or `RS[c] - PS[i,c]` (inverse: ERA, WHIP). Under `"clip"`, floor at 0. Under `"negative"`, retain sign.

2. **Pool denominator.** Three modes:
   - `"ip_weighted"` (default): volume-weighted colSum — `Pool[ERA] = sum(contrib * IP/mean_IP)`. No SGP infrastructure required.
   - `"pool_average"` (Zola extras): endogenous baseline — `ex_ERA = IP * (mean_pool_ERA - player_ERA) / 9`; Pool = sum of positive extras.
   - `"fixed_baseline"`: same extras formula with user-supplied constants via `baseline` argument.

3. **Proportional shares.** `pvm[i,c] = contrib_vol[i,c] / Pool[c]`. Under `"clip"`: sum to 1.0 per category within 1e-10. Under `"negative"`: positive shares sum to 1.0.

4. **Category-weighted total.** `total_pvm[i] = pvm_mat %*% cat_weights`. Cross-role NAs zero-filled before multiply.

**Critical design choices:**
- `hitter_split` in conceptual spec = `config$budget_split` in the actual repo (0.60 default, not 67/33).
- `rate_pool = "fixed_baseline"` requires user-supplied `baseline` — `league_config()` has no ERA/WHIP/AVG constants. Abort class: `rotostats_error_missing_config_field`.
- `baseline` supplied under non-`fixed_baseline` mode: silent no-op.
- `multi_pos = "all"` guard required (pattern from `zar.R:176–191`).
- `cat_pct` validation order: type → names coverage (`rotostats_error_category_mismatch`) → sum (`rotostats_error_cat_pct_sum`).
- OPS: forward-compatible code path, no sign flip, AB volume weighting, not tested this run.

**From `test-spec.md` — key test scenarios and tolerances:**
- TS-PVM-3: `sum(pvm_[CAT]) == 1.0` within atol=1e-10 for all 10 categories (clip mode).
- TS-PVM-4: `sum(positive pvm_[CAT]) == 1.0` within 1e-10 (negative mode); negative values present.
- TS-PVM-5: boundary player `pvm_[CAT] ~= 0`; `total_pvm[boundary] ~= 0`.
- TS-PVM-12/13: `total_pvm` matches CAT%-weighted analytic formula (atol=1e-12).
- PROP-3: `total_pvm` equals manual weighted sum (NA → 0) at atol=1e-12.

**From `sim-spec.md` — 4 Monte Carlo studies:**

### Problems Encountered and Resolutions

#### Block Cycle 1 — DEFECT-1 (builder): `total_pvm` all-NA

**Signal:** BLOCK from tester (2026-04-23 23:15 UTC).

**Root cause:** `pvm_mat` contains NA values for cross-role player-category pairs (pitchers have NA for hitter stats; hitters have NA for pitcher stats). The matrix multiply `pvm_mat %*% cw_vec` in Step 9 propagated these NAs, producing `total_pvm = NA` for every row.

**Fix (commit e07a934):** Before the matrix multiply, copy `pvm_mat` into `pvm_mat_for_total` and zero-fill all NA values. The original `pvm_mat` (with NAs preserved) is used for the output `pvm_[CAT]` columns.

**Tests fixed:** TS-PVM-5, TS-PVM-12, TS-PVM-13, PROP-3 (all from all-NA `total_pvm`).

#### Block Cycle 1 — DEFECT-2 (simulator): DGP `position_assignments` keyed by player name

**Signal:** BLOCK from tester (2026-04-23 23:15 UTC, same cycle as DEFECT-1) and prior HOLD from simulator (2026-04-23 23:01–02 UTC).

**Root cause:** `dgp_pvm.R` constructed `position_assignments` keyed by `player_name` (e.g., `"3B_H_8"`). `pvm()` looks up positions via `projections$PLAYER_ID` (uppercase integer), so the lookup found no matches. Result: 0 rostered players, `rotostats_error_zero_pool` in 100% of replications.

**Fix (commit c7ea5c2):** Changed DGP `position_assignments` to be keyed by `PLAYER_ID` (uppercase integer coerced to character), matching `pvm()`'s `as.character(projections$PLAYER_ID)` lookup.

**Outcome:** Simulation pre-check gate passed (Study 1 error rate: 1.0 → 0.0).

#### Block Cycle 2 — DEFECT-3 (builder): `rate_pool = "pool_average"` zero-pool for rate stats

**Signal:** BLOCK from tester (2026-04-24 08:45 UTC) after Study 2 pre-check failure (max_error_rate = 0.9 for pool_average scenarios).

**Root cause:** For IP-denominated stats (ERA, WHIP) under `pool_average`, the pool formula was `Pool = sum(ex[contrib > 0])`. Since `mean_pool_ERA` is the count-weighted mean of above-replacement ERA values, the sum `IP * (mean_pool_ERA - ERA[i])` over all above-replacement pitchers algebraically cancels to zero (identity of mean). With near-uniform IP (DGP construction), Pool ≈ 0, triggering `rotostats_error_zero_pool` in ~85–93% of replications.

**Fix (commit ca6ecbf):** Changed `Pool[cat] <- sum(ex[contrib[, cat] > 0])` to `Pool[cat] <- sum(pmax(ex, 0), na.rm = TRUE)` for both IP-denominated and AB-denominated rate stats under `pool_average`. `contrib_vol` is set to `pmax(ex, 0)` under `"clip"` so numerator sums equal Pool, guaranteeing sum-to-1. Under `"negative"`, `contrib_vol = ex` (raw), and positive pvm values still sum to 1 by the same identity.

#### Block Cycle 2 — DEFECT-4 (builder): `rate_pool = "fixed_baseline"` violates sum-to-1

**Signal:** Same BLOCK cycle as DEFECT-3.

**Root cause:** Same structural bug in the `fixed_baseline` branch. Above-replacement pitchers with ERA in `(baseline_ERA, RS_ERA)` — worse than the fixed baseline but still above replacement — had negative extras but positive contrib. The old formula `sum(ex[contrib > 0])` included these negative extras in Pool, causing heavy cancellation. With partial cancellation, Pool was non-zero but tiny, making individual pvm values enormous (sum deviation up to 1836).

**Fix (commit ca6ecbf):** Same correction — `Pool[cat] <- sum(pmax(ex, 0), na.rm = TRUE)` for all rate stats under `fixed_baseline`. After fix: max sum deviation = 2.00e-15.

## Design Decisions

1. **RS[c] computation**: Position-slot-weighted mean of `replacement$replacement_stats`, separately for hitter and pitcher categories. This is the "preferred approach" from spec.md §Step 1. Fallback to unweighted mean when no matching positions appear in replacement stats.

2. **`pool_average` Pool formula**: Pool = `sum(pmax(ex, 0))` rather than `sum(ex[contrib > 0])`. This prevents the algebraic cancellation that occurs when above-replacement players are distributed symmetrically around the pool mean. The invariant `sum(pmax(ex[i], 0)) / Pool = 1` is preserved exactly. This was the key fix in Block Cycle 2 (DEFECT-3 and DEFECT-4).

3. **NA-zero-fill for `total_pvm`**: A working copy of `pvm_mat` with NAs zeroed is used exclusively for the matrix multiply. The output `pvm_[CAT]` columns retain NAs for non-participating players. This preserves the diagnostic value of NAs (e.g., a pitcher correctly shows NA for HR) while ensuring `total_pvm` is always finite (Block Cycle 1 fix, DEFECT-1).

4. **`baseline` validation**: Names in `baseline` are uppercased at entry time. Names not in scored categories abort with `rotostats_error_category_mismatch`. The `fixed_baseline` completeness check (Step 2) fires after this and checks that every scored rate stat has a baseline entry — aborting with `rotostats_error_missing_config_field` if any is missing.

5. **`hitter_split` → `budget_split` mapping**: The spec used the conceptual name `hitter_split`; the implementation reads `config$budget_split`. This was resolved by the planner in comprehension.md. No new field was added to `league_config()`.

6. **`devtools::document()` workaround**: `devtools::load_all()` fails due to `R/sgp.R` top-level execution. `roxygen2::roxygenise()` with a custom `load_code` function (wrapping `sgp.R` in `tryCatch`) was used to generate `man/pvm.Rd` and update `NAMESPACE` successfully.
